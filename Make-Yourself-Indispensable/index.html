3 Key Goals

There are 3 main key points that you should keep in mind all the time. If you maximize this 3 factors I garantee that you'll have a long and boring career at your company

Maximize Entropy

I'm talking, of course, about the right definition of Entropy: the amount of chaos in a system. How much information every piece of that system gives about the whole system. Systems tend to increase it's internal disorder. You have to do a conscious effort to decrease the entropy of a system. Ok, don't do it! In fact, if you're going to put effort on anything (and I'm almost always against putting effort to things) you have to try to increase entropy. You will find that you have a lot to gain in a chaotic system. Everything will more difficult to understand, and learn one thing won't give you points to learn another part of the system.

Write a LOT of code

Ok, that's the 101. If you write a lot of code your name will be everywhere and people will start thinking that you are an important person inside the organization Also, more code means more entropy. So whenever possible, you should write more, not less code

Duplicate

An obvious way to generate more code is to duplicate.

Duplycate

A better way is to duplicate, but with minor changes everywhere. That way, nobody will be really sure what the code is actually doing and they will be really affraid to change your code. You will keep the ownership ;) More chaos, more fear, we like it

Tests

Not writing tests is not an option. Tests are code, and like I said, you should write a lot of code. But you can always write tests in a way that they don't really make anyone's life easier. Test your already-tested dependencies, for example. Or make tests so heavy that no one would like to run them Also, if you learn to make a lot of tests that could break without meaning that the code is broken everyone will be a bit more scared all the time. Be creative there. One tip: try tests that depend on time so they can fail randomly. That leds me to the next point
